## 2. Architecture Overview

```
Catalog .txt files
      │
      ▼  ingest.py
RecursiveCharacterTextSplitter (600 chars, 120 overlap)
      │
      ▼
HuggingFace Embeddings (all-MiniLM-L6-v2, local, free)
      │
      ▼
FAISS Vector Store (cosine similarity)
      │
      │  rag_chain.py (query time)
      ▼
FAISS Retriever (k=6)  ←── User Query
      │
      ▼
Structured System Prompt (enforce citations + abstention)
      │
      ▼
Groq LLM (llama3-8b-8192, temp=0, free)
      │
      ▼
Structured Response: Answer | Why | Citations | Questions | Assumptions
```

## 3. Chunking & Retrieval Choices and Tradeoffs

**Chunk size: 600 characters**  
Most MIT course descriptions (name + prereqs + description) fit in ~400-600 chars. This ensures one course block per chunk, making citation accurate.

**Overlap: 120 characters (20%)**  
Prevents prerequisite conditions that span paragraph boundaries from being lost at chunk edges.

**Separators**: `\n---\n` → `\n\n` → `\n` → space  
The `---` delimiter separates course blocks in the source files. Splitting on it first keeps course data atomic.

**Retriever**: top-k=6, cosine similarity  
6 chunks provides enough context for multi-hop prerequisite chains (typically 3-4 courses in a chain) without exceeding the LLM's context window or diluting relevance.

**Tradeoff**: Larger chunks reduce recall precision but improve completeness. k=6 was chosen over k=10 to avoid injecting irrelevant course data that confuses the LLM.

## 4. Prompts and Agent Roles (High Level)

The system uses a **single LangChain RAG chain** (retriever → prompt → LLM) with a carefully engineered system prompt.

**System prompt enforces**:
- Never state facts not in the retrieved context
- Always include `[Source: filename | Section: section]` citations
- Use the fixed 5-section output format
- If information is absent: output "I don't have that information in the provided catalog/policies" and suggest next steps

**Equivalent to agentic stages**:
- *Intake*: Streamlit UI gathers question context; clarifying questions are generated by the LLM itself in the "Clarifying Questions" section
- *Retrieval*: FAISS retriever with source metadata extraction
- *Planning/Answering*: Groq LLM with structured prompt
- *Verification*: Keyword + citation + abstention scoring in evaluate.py (post-hoc)

## 5. Evaluation Summary, Failure Modes, and Next Improvements

**Test set**: 25 queries across 4 categories  
- 10 prerequisite checks (eligible / not eligible)  
- 5 prerequisite chain questions (multi-hop)  
- 5 program requirement questions  
- 5 not-in-docs / trick questions  

**Expected metrics**:
| Metric | Score |
|---|---|
| Citation coverage rate | ≥ 90% |
| Abstention accuracy | ≥ 80% |
| Structured format compliance | ≥ 95% |
| Overall pass rate | ≥ 80% |

**Key failure modes**:
1. **Availability questions** — The catalog notes historical semester patterns, but actual term schedules aren't in the documents. The system correctly abstains here.
2. **Instructor override details** — The policy acknowledges overrides exist but doesn't detail step-by-step approval. Partial abstention is the correct behavior.
3. **Multi-hop chain completeness** — Occasionally the LLM misses an intermediate step (e.g., omitting 18.06 in the 6.3900 chain). Re-ranking or a dedicated chain-of-thought prompt would help.

**Next improvements**:
1. Add a **Verifier chain** that re-reads the response and checks each claim against retrieved chunks before returning
2. Ingest **live term schedules** (MIT Subject Listing HTML) to answer availability questions without abstaining
3. **Cross-encoder re-ranking** (e.g., `ms-marco-MiniLM-L-6-v2`) to improve chunk relevance beyond cosine similarity
4. Switch to **`llama3-70b-8192`** (still free on Groq, larger model) for better multi-hop reasoning
